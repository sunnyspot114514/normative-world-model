# Verdict

**PASS_WITH_FIXES.**

Both Stage-2 slices are local-only, purely additive, and sound in their core mechanisms: every authorization/confirmation/lock field is closed and machine-verified closed; the selector is deterministic, strata-balanced, exclusion-safe, recomputes hard policy through the deterministic oracle rather than inferring all-reject, and stops without relaxation; the payload verifier is exact-set, escape-proof (verified empirically, including a Windows directory-junction attack), and fails closed on undeclared/missing files; the publisher weight plan is index-driven with no hard-coded shard count. No network, weight-download, GPU, server-execution, confirmation-access, or real-corpus entry point exists in either module.

Three bounded gaps must land and receive Codex counter-review before restricted public-metadata downloader work begins: (B1) the contract validator does not guard several fields its own document claims it guards, including the two downloader-protective source fields, and its token-budget check passes vacuously when keys are deleted; (B2) the weight-plan resolver consumes a normalized sibling shape that raw Hugging Face API metadata does not have, and the normalization is unspecified; (B3) the tokenizer-package comparison ignores tokenization-relevant sections (`truncation`, `padding`) and added-token behavioral flags, while the Stage-2 document claims an "exact" preprocessing comparison. None of these crosses an authorization or data boundary; all are small, reviewable fixes.

This verdict authorizes neither real-source selection, model weights, GPU rental, project-scenario inference, confirmation, nor science. Downloader *execution* (any network write) additionally requires the minimum controls enumerated under "Downloader entry decision" to be committed first; the Stage-2 document already forbids execution until the dedicated code path and manifest tests exist.

**B2. The publisher weight-plan resolver's expected sibling shape does not exist in raw Hugging Face metadata, and the normalization is unspecified.**
Evidence: `src/normative_world_model/phase5_serialization.py:255-264` reads flat `sibling["size"]` and `sibling["lfs_sha256"]`. Real HF API siblings (`/api/models/{repo}?blobs=true` at a pinned revision) carry `{"rfilename", "size", "lfs": {"oid": "<sha256>", "size": ...}}`. Impact: on the downloader path, an improvised adapter is precisely where hash semantics can silently shift (e.g., `oid` vs blob SHA-256 confusion). Smallest correct fix: commit the normalization contract ŌĆö `rfilename` pass-through, `size` ŌåÉ sibling `size`, `lfs_sha256` ŌåÉ `lfs.oid`, reject siblings missing either, reject non-lowercase/non-64-hex ŌĆö with fixture tests against realistic raw API shapes, before any manifest construction.

**B3. Tokenizer-package comparison omits tokenization-relevant sections and added-token attributes; the document's "exact ŌĆ” preprocessing comparison" claim is stronger than the code.**
Evidence: `src/normative_world_model/phase5_serialization.py:31-37` compares only `model`, `normalizer`, `pre_tokenizer`, `post_processor`, `decoder`; `phase5_serialization.py:105-114` compares added tokens only as `id ŌåÆ content`, ignoring `special`, `normalized`, `lstrip`, `rstrip`, `single_word`. Impact: truncation/padding/added-token-flag differences change token-ID sequences; the per-prompt equality proof would catch them only for locked prompts that actually exercise the difference (e.g., only prompts exceeding a truncation limit), and the protocol's freeze-time requirement is "identical core vocabulary and preprocessing" (`docs/PHASE5_SCALE_INFERENCE_PROTOCOL_DRAFT.md:53`). The live metadata at the pinned revisions was independently verified last round to have identical normalizer/pre-tokenizer/decoder/post-processor and identical shared added tokens, so there is no live discrepancy today ŌĆö this is a screener-completeness gap against the document's claim, not a detected tokenizer difference. Smallest correct fix: extend comparison to `truncation` and `padding`, compare full added-token attribute dicts (not just content), and hash/compare every downloaded metadata file including `chat_template.jinja`; or narrow the document claim to "core-vocabulary and five-section preprocessing screen, with the all-prompt token-ID proof as the authoritative check".

# Non-blocking findings

1. **A caller can falsely label project-derived content as public synthetic; hard links make it concrete.** `verify_remote_payload` copies the caller-supplied content class into the manifest without any provenance check (`phase5_preflight.py:429-435, 468-475`). Probe: a hard link (`os.link`) into the payload root pointing at a project-content file, declared as `public_synthetic_inputs`, passes verification with `st_nlink = 2`. The document's claim is literally only about undeclared files and class membership, so code matches doc ŌĆö but the audit question's answer is *yes, mislabeling passes*. 2. **Proof rows do not bind the tokenizer snapshots used.** `prove_common_prompt_token_equality` rows contain only `prompt_id`, `prompt_text_sha256`, `token_count`, `token_ids` (`phase5_serialization.py:206-213`); the tokenizer objects are caller-supplied, so the proof artifact alone does not establish *which* tokenizer bytes produced the IDs. This is consistent with `STOP_WITHOUT_RELAXATION` ŌĆö it stops rather than relaxes ŌĆö but means one bad real family blocks selection with no exclusion path short of editing the exclusion list.
