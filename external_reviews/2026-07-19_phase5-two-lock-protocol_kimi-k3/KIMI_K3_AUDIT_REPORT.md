# Independent audit: Phase-5 two-lock protocol at `e9da3e3`

## Blocking findings

Each is a bounded wording/specification fix; none requires a design change.

1. **Remote-payload allowlist does not cover lossless derived encodings of prompt text, and the served-snapshot re-proof's execution location is unstated.** The protocol and TOML exclude "project prompt text" and "scientific request bodies" from the synthetic rental (`configs/phase5_scale_inference_draft.toml:138-143`, `docs/PHASE5_SCALE_INFERENCE_PROTOCOL_DRAFT.md:91`), and Lock A simultaneously requires an all-common-prompt token-ID proof (`…PROTOCOL_DRAFT.md:159`). Token-ID sequences of locked prompts decode losslessly back to prompt text with the base tokenizer, yet they are not named by any exclusion category, and the protocol does not state whether Lock B's served-snapshot re-proof (`…PROTOCOL_DRAFT.md:55`) runs remotely (which would pull locked prompts or their token IDs onto the rental) or locally on transferred tokenizer snapshots. Fix: extend the fail-closed allowlist to exclude any artifact that losslessly encodes selected prompt text (explicitly including locked-prompt token-ID sequences), and state that the served-snapshot re-proof executes locally from rehashed, transferred tokenizer snapshots — or explicitly allowlist and hash a token-ID-only remote payload.

2. **Stability-probe counting and pass predicate are under-specified.** The protocol says the grid `[1, 8, 16, 32]` is tested "with eight stability requests per checkpoint and mode" (`…PROTOCOL_DRAFT.md:96`), readable as 8 requests per checkpoint-by-mode condition across the whole grid, while the TOML says `stability_requests_per_candidate_per_condition = 8` (i.e., 8 × 4 candidates × 4 conditions = 128 requests). Neither file defines what makes a stability probe "pass" (completion without crash and without post-retry 5xx? schema validity? a latency bound?). The selected-concurrency rule ("largest candidate passing all four checkpoint-by-mode probes") is deterministic only once that predicate exists. Fix: harmonize the count wording to per-candidate-per-condition and predeclare the exact pass predicate.

3. **The per-checkpoint one-GPU-hour cap's scope is undefined and the warm-up block exists only in the TOML.** Two mode-conditions × the 30-minute measurement cap already sum to exactly one GPU-hour, leaving zero room inside the cap for grid stability probing, warm-up, or the protocol-shaped component (which has no time cap at all). It is unstated whether "total measured time per checkpoint" includes those activities; read inclusively the budget is infeasible, read exclusively the protocol-shaped component is uncapped. Separately, `warmup_concurrency_blocks = 1` (`…toml:157`, implementing K3's "exclude one warm-up block") has no counterpart in the Markdown protocol. Fix: define which activities count against the per-condition and per-checkpoint caps, and add the warm-up exclusion to the protocol text.

## Throughput and serialization adjudication

- **Protocol-shaped vs decode-ceiling separation:** sound. The protocol-shaped component (16 requests per checkpoint-by-mode-by-input-length cell at 1,024/3,072/6,000 total rendered prompt tokens) measures overhead, schema-path stability, and latency; the decode-ceiling component (1,800-token target outputs) measures sustained decode. The long-output schema is explicitly barred from any behavioral or schema-quality claim, and `quality_metrics_allowed = false`.
- **Budget:** 6,000 + 2,048 ≤ 8,192 verified; the "2,192 tokens remaining" statement is exact.
- **Grid and selection rule:** `[1, 8, 16, 32]` with "largest candidate passing all four checkpoint-by-mode probes" is deterministic and matches K3's both-checkpoints-same-rental requirement; out-of-grid values and rule changes require Lock-A amendment plus delta review.
- **Windows and CV:** three windows per condition, each ≥ 8,192 generated tokens, CV ≤ 0.20 across windows — this closes Codex's "slow condition times out with too little evidence" hole.
- **Cost formula:** fixed overhead (both downloads, publisher-hash verification, load/unload, JIT/warm-up, packaging/transfer) plus worst-case variable projection (all 3,072 requests plus the frozen retry allowance), × 1.5; expected and worst-case projections reported separately; engineering ceiling only.
- **Evidence minimum:** an unmeasured cell or unstable condition cannot pass — both checkpoints and both modes must be valid; the selected concurrency must pass all four probes; a crash, a post-retry failure, or cap exhaustion before the minimum evidence makes the smoke insufficient for Lock B (`require_both_checkpoints_expensive_condition = true`). The residual weaknesses are the undefined stability-pass predicate and the cap-scope ambiguity (blocking findings 2–3); CV over exactly three windows is noisy but predeclared.
- **Serialization proof:** the three-stage design (synthetic checker probe before derivation, all locked common prompts after derivation, served-snapshot re-proof before Lock B, any mismatch stops the common mode) is adequate and is faithfully encoded in `[serialization_proof]`; the live metadata underlying it was independently re-verified above.
